# Java: feeding a dependency SBOM via `--sbom`

`sbom-quality` bundles no Java toolchain. For Java you **bring your own** CycloneDX
dependency SBOM, generated by your build, and hand it to the pipeline with `--sbom`
(mutually exclusive with `--go-mod`; `--image` optional). The pipeline then enriches,
augments, quality-patches, and validates it into one CycloneDX 1.6 document — the
same path Go modules take, so any future ecosystem (Python, npm, …) is a docs + CI
job, not a code change.

Recommended: the **native build plugins** (`cyclonedx-maven` / `cyclonedx-gradle`).
They emit native 1.6 with per-component hashes and run inside the `mvn`/`gradle`
your CI already invokes. cdxgen is a documented fallback for polyglot builds — see
the bottom of this page.

Measured post-pipeline sbomqs (build plugins): solo `--sbom` ≈ **8.3**; a `--sbom +
--image` merge on a full-OS Jib base ≈ **6.6** (the image adds ~140 OS packages that
dilute coverage — it is additive, app identity comes from the build-source SBOM).

## Maven — `cyclonedx-maven`

The plugin can be invoked fully qualified from the CLI, so no `pom.xml` change is
needed:

```sh
mvn -B org.cyclonedx:cyclonedx-maven-plugin:2.9.1:makeBom -DoutputFormat=json
# -> target/bom.json  (native CycloneDX 1.6)

sbom-quality --sbom target/bom.json --supplier-name "ACME" > sbom.cdx.json
```

GitHub Actions:

```yaml
- uses: actions/setup-java@v4
  with: { distribution: temurin, java-version: "17" }
- name: Build dependency SBOM
  run: mvn -B org.cyclonedx:cyclonedx-maven-plugin:2.9.1:makeBom -DoutputFormat=json
- name: Quality SBOM
  run: |
    docker run --rm -v "$PWD":/work ghcr.io/nissessenap/sbom-quality:latest \
      --sbom /work/target/bom.json --supplier-name "ACME" > sbom.cdx.json
```

## Gradle — `cyclonedx-gradle`

Add the plugin (pinned) and pin the schema to 1.6 (`build.gradle.kts`):

```kotlin
plugins {
    id("org.cyclonedx.bom") version "2.3.1"
}
tasks.cyclonedxBom {
    setSchemaVersion("1.6")
    setOutputFormat("json")
    setIncludeConfigs(listOf("runtimeClasspath"))
}
```

```sh
./gradlew cyclonedxBom
# -> build/reports/bom.json

sbom-quality --sbom build/reports/bom.json --supplier-name "ACME" > sbom.cdx.json
```

> If your Gradle build sets a git `origin` in scp form (`git@host:org/repo`), the
> plugin may emit a non-IRI VCS URL that fails 1.6 validation. GitHub's `actions/checkout`
> uses an https remote (valid), so CI is unaffected; locally, use an https remote.

## Merge with a Jib image

Jib's default base is a full-OS `eclipse-temurin`, so the image scan is a coarse,
noisy second view. Merge it *additively* with the build-source SBOM — image stays
primary (keeps OS packages), the build SBOM supplies the app jars and their hashes:

```sh
./gradlew jibDockerBuild            # loads <image> into the local docker daemon
sbom-quality --image <image> --sbom build/reports/bom.json \
  --supplier-name "ACME" > sbom.cdx.json
```

## cdxgen fallback (polyglot / non-mvn-gradle builds)

[cdxgen](https://github.com/CycloneDX/cdxgen) is one build-agnostic generator that
also covers Python/npm/etc. Use it when the native build plugins don't fit. It works
through `--sbom`, but with trade-offs vs the build plugins:

- **0% component hashes** (build plugins emit ~90–100%) → lower sbomqs on the raw
  generator output (cdxgen ~5.5 vs build-plugin ~6.4–6.8, both pre-pipeline; the
  ~8.3 above is post-pipeline) and weaker component identity.
- **Defaults to CycloneDX 1.7** — pin `--spec-version 1.6` to match the pipeline.
- **Needs Node** on the runner (`actions/setup-node`).

```sh
cdxgen -t java --spec-version 1.6 -o cdxgen.bom.json .
sbom-quality --sbom cdxgen.bom.json --supplier-name "ACME" > sbom.cdx.json
```
