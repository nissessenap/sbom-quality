# Rust: feeding a dependency SBOM via `--sbom`

`sbom-quality` bundles no Rust toolchain. For Rust/Cargo you **bring your own** CycloneDX
dependency SBOM, generated by your build, and hand it to the pipeline with `--sbom`
(mutually exclusive with `--go-mod`; `--image` optional). The pipeline then enriches,
augments, quality-patches, and validates it into one CycloneDX 1.6 document — the same
path Go, Java and npm take, so Rust is a docs + CI job, not a code change.

Two generators are supported, both gated in CI:

| Generator | Spec | Licenses | Best for | Measured solo |
|-----------|------|----------|----------|---------------|
| **`cargo-cyclonedx`** | 1.5 (up-converted to 1.6) | native, offline | Rust-only projects | ≈ **8.2** |
| **`cdxgen -t rust`** | 1.6 native | `FETCH_LICENSE` from crates.io | polyglot / monorepos | ≈ **8.3** |

Both carry SHA-256 hashes, SPDX license expressions and `pkg:cargo` purls; `parlay` then
enriches each component with supplier and VCS references from
[ecosyste.ms](https://ecosyste.ms). sbom-utility CycloneDX 1.6 conformance passes for both.

Pick `cargo-cyclonedx` for a Rust-only build (native tool, licenses without a network
license-fetch, best fidelity). Pick `cdxgen` for polyglot builds where one generator
covers several ecosystems.

## `cargo-cyclonedx` (Rust-native)

cargo-cyclonedx maxes at CycloneDX **1.5** today; the `--sbom` guard accepts 1.5 and
up-converts it to 1.6 (you'll see a one-line stderr notice). It reads licenses straight
from each crate's `Cargo.toml`, so no license-fetch flag is needed.

```sh
cargo install cargo-cyclonedx           # --locked if your rustc is older stable
cargo cyclonedx --spec-version 1.5 --format json
# -> <crate>.cdx.json  (CycloneDX 1.5: SHA-256 hashes, SPDX expressions, pkg:cargo purls)

sbom-quality --sbom <crate>.cdx.json --supplier-name "ACME" > sbom.cdx.json
```

> Once cargo-cyclonedx ships CycloneDX 1.6 (upstream
> [cyclonedx-rust-cargo#769](https://github.com/CycloneDX/cyclonedx-rust-cargo/issues/769))
> the up-convert becomes a no-op — nothing else changes. (CycloneDX 1.5 support itself
> landed via the now-closed [#646](https://github.com/CycloneDX/cyclonedx-rust-cargo/issues/646).)

## `cdxgen -t rust` (polyglot)

[cdxgen](https://github.com/CycloneDX/cdxgen) reads `Cargo.lock`, so run it after a
resolve (`cargo generate-lockfile` or any build). Pin `--spec-version 1.6` (cdxgen's
default is 1.7, which the pipeline down-converts anyway):

```sh
cargo generate-lockfile                       # ensure Cargo.lock exists
FETCH_LICENSE=true cdxgen -t rust --spec-version 1.6 -o bom.json .
# -> bom.json  (CycloneDX 1.6: SHA-256 hashes, SPDX license expressions, pkg:cargo purls)

sbom-quality --sbom bom.json --supplier-name "ACME" > sbom.cdx.json
```

`FETCH_LICENSE=true` is what pulls each crate's SPDX license from crates.io — licenses
aren't in `Cargo.lock`, so without it the components carry hashes and purls but no
licenses. It needs network to crates.io.

GitHub Actions (cdxgen):

```yaml
- uses: actions/setup-node@v4          # cdxgen is a Node CLI
  with: { node-version: "22" }
- name: Build dependency SBOM
  run: |
    npm install -g @cyclonedx/cdxgen
    FETCH_LICENSE=true cdxgen -t rust --spec-version 1.6 -o bom.json .
- name: Quality SBOM
  run: |
    docker run --rm -v "$PWD":/work ghcr.io/nissessenap/sbom-quality:latest \
      --sbom /work/bom.json --supplier-name "ACME" > sbom.cdx.json
```

> **license expressions:** Rust crates commonly license as SPDX expressions
> (`MIT OR Apache-2.0`). CycloneDX 1.6 rejects non-enum ids inside an expression, so the
> quality-patch stage unwraps each expression to a free-form declared `license.name` — the
> string is preserved verbatim and credited, never dropped. Nothing to configure.
